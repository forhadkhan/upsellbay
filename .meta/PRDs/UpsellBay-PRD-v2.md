# **PART A · Market Research & Competitive Landscape**

## **1\. Market Positioning & The Gap**

The WooCommerce AOV-optimisation market is dominated by full funnel-builder suites that require merchants to replace their checkout template as a precondition of using order bumps. This creates deep plugin lock-in, theme conflicts, and a steep learning curve - all of which are documented pain points in 2026 support forums and reviews. The strategic gap UpsellBay fills is narrow but concrete: a lightweight, additive order bump and cross-sell plugin that works natively on top of any existing WooCommerce checkout, requires no checkout replacement, and is listed on the WooCommerce Extensions Marketplace as a first-class citizen.

| **Market signal 1** | **Funnel builder fatigue is real.** The most common complaint in FunnelKit and CartFlows reviews in 2026 is checkout conflicts and layout breaks when run alongside other plugins. Stores want AOV gains without architectural risk.                                                                                                   |
| ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Market signal 2** | **UpsellWP validates the 'upsells-only' positioning.** UpsellWP is explicitly described by reviewers as 'Best for Upsells-Only (No Funnels)' - confirming that a focused, non-funnel-builder product has a defined, willing audience.                                                                                                  |
| **Market signal 3** | **Block Checkout is now the default.** WooCommerce 10.x makes Block Checkout the default for all new installs. Most existing order bump plugins use PHP hooks that do not fire in the block context. HPOS-native, Block-compatible support is a genuine technical moat for early movers.                                               |
| **Market signal 4** | **Post-purchase 1-click tokenisation is unreliable in practice.** Multiple competitors advertise this feature but support threads show frequent failures on WooPayments, Square, and regional gateways. Avoiding tokenisation and using a standard secondary checkout path is more reliable and differentiates UpsellBay on stability. |
| **Price gap**       | **\$69-\$79/yr tier is underpopulated.** FunnelKit starts at \$179/yr, CartFlows at \$189/yr for full features. UpsellWP's Pro is \$69/yr but has no Marketplace listing and limited HPOS documentation. ## **3\. Strategic Differentiation**

**Core wedge statement:** "The fastest way to increase WooCommerce average order value without replacing your checkout, hiring a developer, or paying for a funnel builder."

Three non-negotiable differentiators that must be preserved through every design and engineering decision:

- **Additive, never replacement.** UpsellBay works on top of whatever checkout the store already has. It does not require a custom checkout template. This is the #1 reason stores avoid FunnelKit and CartFlows.
- **Block Checkout native from day one.** This is a technical moat. Most competitors have not invested in the WooCommerce Additional Checkout Fields API. UpsellBay ships with it as a P0 requirement.
- **Reliability over features.** No tokenisation risk. No gateway compatibility gambles. The thank-you page offer uses a standard secondary checkout that works with every WooCommerce-compatible gateway - not just Stripe and PayPal.

# **PART B · UpsellBay PRD v4.0 - Engineering Specification**

## **4\. Plugin Overview**

UpsellBay is a focused, additive average-order-value engine for WooCommerce. It injects highly relevant product offers at three key moments - on the product page, at checkout, and on the thank-you page - without replacing the checkout template, creating custom database tables, or depending on payment gateway tokenisation. All offer configuration is stored in a Custom Post Type. All conversion attribution is stored in WooCommerce order line item meta. The plugin is fully HPOS-compatible and designed to pass the WooCommerce Quality Insights Toolkit (QIT) test suite on first submission.

## **8\. Architecture & Data Design**

### **8.1 No Custom Database Tables**

The v1 PRD proposed a custom table for offer impressions and conversions. This design is rejected. All data is stored in WordPress and WooCommerce native structures. The full data model is:

| **Data entity**                    | **Storage mechanism**                              | **Detail**                                                                                                                                                                                                                     |
| ---------------------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Offer configuration                | Private CPT: upsellbay_offer                       | One CPT post per offer. Post meta stores the full offer schema (see 8.2). CPT gives revision history, familiar WP edit UI, REST API support. Registered with show_in_menu: false - visible only under WooCommerce → UpsellBay. |
| Conversion attribution             | WC order line item meta                            | \_ub_offer_id, \_ub_offer_type, \_ub_discount_applied written to the accepted product's line item via \$item->add_meta_data(). HPOS-safe. Queryable via WC line item API.                                                      |
| Impression counter (live)          | WP transient: ub*impr*{offer_id}\_{YYYYMMDD}       | Integer counter, TTL 25 hours. Incremented on each offer render. No DB write per impression - atomic increment via get_transient / set_transient.                                                                              |
| Monthly analytics rollup           | Post meta on upsellbay*offer: \_ub_stats*{YYYYMM}  | Serialised array {views, accepts, revenue}. Written by daily Action Scheduler rollup job. Source for analytics dashboard - avoids live COUNT(\*) on order tables.                                                              |
| Plugin settings                    | wp_options: upsellbay_settings                     | Single serialised array. Placement defaults, styling tokens, licence key (masked in UI), per-placement enable/disable flags.                                                                                                   |
| Offer priority / display order     | menu_order on upsellbay_offer CPT                  | WP native integer field. Sortable via drag-and-drop in CPT list table. Lower = higher priority.                                                                                                                                |
| Follow-on order source attribution | WC order meta: \_ub_source_order_id, \_ub_offer_id | Written to the thank-you-page follow-on order at creation. Links back to the primary order for reporting.                                                                                                                      |
| Offer session dismissal            | WC session: ub*dismissed*{offer_id}                | Guest-session flag. If shopper clicks 'No thanks' on a bump, do not show the same offer again in this session. Cleared on order completion.                                                                                    |

### **8.2 upsellbay_offer CPT Post Meta Schema**

Every upsellbay_offer post stores the following meta keys. This schema is the single source of truth for both the admin UI and the runtime rule evaluator.

| **Meta key**        | **Type** | **Description**                                                                                                                                                                     |
| ------------------- | -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| \_ub_offer_type     | string   | checkout_bump \| cart_crosssell \| product_upsell \| thankyou_offer                                                                                                                 |
| \_ub_product_id     | int      | WooCommerce product ID to display. Validates against wc_get_product() on save.                                                                                                      |
| \_ub_discount_type  | string   | percent \| fixed \| none                                                                                                                                                            |
| \_ub_discount_value | float    | Discount amount for the offer. 0.0 if \_ub_discount_type is 'none'.                                                                                                                 |
| \_ub_headline       | string   | Short offer headline. Max 80 chars. Sanitised with sanitize_text_field().                                                                                                           |
| \_ub_body           | string   | Offer body copy. Max 200 chars. Sanitised with wp_kses_post().                                                                                                                      |
| \_ub_rules          | array    | Targeting rules. Each rule: {field, operator, value}. Fields: cart_product_id \| cart_category_id \| cart_total \| user_role. Operators: is \| is_not \| greater_than \| less_than. |
| \_ub_rules_match    | string   | all (AND) \| any (OR). Default: all.                                                                                                                                                |
| \_ub_placement      | string   | For checkout_bump: before_payment \| after_review. For thankyou_offer: above_details \| below_details.                                                                              |
| \_ub_show_image     | bool     | Whether to display product thumbnail in the offer widget. Default: true.                                                                                                            |
| \_ub_active         | bool     | Whether offer is live. Paused offers are skipped by the rule evaluator.                                                                                                             |
| \_ub*stats*{YYYYMM} | array    | Monthly rollup: {views:int, accepts:int, revenue:float}. Written by AS daily job.                                                                                                   |

### **8.3 Discount Application Strategy**

UpsellBay applies offer discounts via a session-scoped WooCommerce cart fee, not a persistent WooCommerce coupon. This prevents coupon table pollution at scale and eliminates coupon code leakage risk (shoppers sharing codes externally).

- **Mechanism:** When the bump is accepted, a negative cart fee line item is added via woocommerce_cart_calculate_fees. The fee is scoped to the bumped product only (calculated as discount_type × discount_value × product_price).
- **Persistence:** Fee is stored in WC session alongside the cart. It is cleared when the cart is emptied, the bump is unchecked, or the order is placed - no cleanup job required.
- **Attribution:** The discount amount is written to order line item meta (\_ub_discount_applied) for analytics and reporting.
- **WooCommerce Subscriptions safeguard:** If the offer product (or any existing cart item that may be affected by fees) is a WooCommerce Subscription product (class WC_Product_Subscription or WC_Product_Subscription_Variation), the cart fee discount is NOT applied. A note is shown to the shopper: 'Discount applies to regular products only.' This prevents the discount from being inherited by recurring renewal orders.

### **8.4 Impression & Conversion Tracking**

Impressions are tracked per-offer per-day using WP transients (no DB write per page load). A daily Action Scheduler job rolls transient counters into monthly post meta. Conversions are captured directly on woocommerce_checkout_create_order_line_item. There is no custom table at any stage.

- **Impression write:** On offer render, increment ub*impr*{offer_id}\_{YYYYMMDD} transient. Key TTL: 25 hours. If transient does not exist, create with value 1.
- **Daily rollup job:** Action Scheduler job ub*daily_rollup (fires at 02:00 site time) reads all ub_impr*\* transients, aggregates into \_ub*stats*{YYYYMM} post meta on the corresponding offer post, and deletes processed transients.
- **Attribution write:** woocommerce_checkout_create_order_line_item callback writes \_ub_offer_id, \_ub_offer_type, and \_ub_discount_applied to the accepted product's WC_Order_Item_Product via add_meta_data().
- **Double-count guard:** Before writing attribution meta, check if \_ub_offer_id already exists on the line item. If yes, skip. This prevents duplicate attribution if woocommerce_checkout_create_order_line_item fires more than once for the same item (known edge case in some gateway flows).

## **11\. Key User Flows**

### **Flow 1 - Admin Creates a Checkout Order Bump**

- Admin navigates to WooCommerce → UpsellBay → Add New Offer.
- Admin selects Offer Type: Checkout Bump. Page reloads the form for the checkout_bump schema.
- Admin selects offer product (e.g., 'Leather Care Kit') via Select2 AJAX product search.
- Admin sets discount: 15% off. Form shows calculated discounted price for confirmation.
- Admin adds targeting rule: 'Cart contains category: Shoes'. Selects AND/OR logic.
- Admin enters Headline: 'Protect your new shoes' and body copy.
- Admin saves. Offer post published with status active. Offer immediately live on checkout.
- Admin enables test mode and places a test order with shoes in cart. Confirms bump renders correctly.
- Admin disables test mode. Bump goes live for real shoppers.

### **Flow 2 - Shopper Accepts Checkout Bump (Classic Checkout)**

- Shopper adds Running Shoes to cart, navigates to /checkout/.
- UpsellBay rule evaluator runs at woocommerce_review_order_before_submit. Evaluates all active checkout_bump offers in priority order. Leather Care Kit offer passes (cart contains Shoes category). Offer HTML injected.
- Shopper sees: \[ ☐ \] Add Leather Care Kit - normally £15, yours for £12.75 today.
- Shopper checks the checkbox. JS POSTs to /wp-json/upsellbay/v1/bump-toggle {product_id, offer_id, action:'add'}.
- REST handler validates request, adds product to cart at discounted price via wc_add_to_cart(). Negative cart fee applied via woocommerce_cart_calculate_fees. Returns {success:true, cart_totals:...}.
- JS triggers WC update_checkout to refresh order totals. Updated totals displayed.
- Shopper completes purchase. woocommerce_checkout_create_order_line_item fires. UpsellBay writes \_ub_offer_id and \_ub_offer_type = checkout_bump to the Leather Care Kit line item. Impression and conversion recorded.

### **Flow 3 - Shopper Accepts Thank-You Page Offer**

- Shopper completes primary checkout for Running Shoes. woocommerce_thankyou fires.
- UpsellBay evaluates active thankyou_offer rules against the completed order's line items. Matching offer (e.g., matching socks for £8) is rendered on the order-received page.
- Shopper clicks 'Add to order'. JS adds product to a new WC cart and redirects to /checkout/?ub_follow_on=1. The primary order's ID is saved to WC session as \_ub_source_order_id.
- On the follow-on checkout, woocommerce_checkout_get_value pre-fills billing/shipping fields from the source order. Shopper reviews the single-product cart and proceeds.
- Shopper places the second order through their payment gateway of choice. Order created with \_ub_source_order_id and \_ub_offer_id order meta. Attribution written to line item.
- If secondary payment fails: second order has status 'failed'. Primary order is completely unaffected - no status mutation, no held state.

### **12.2 Block Checkout Integration Detail**

This is a committed implementation approach, not a spike. The WooCommerce Additional Checkout Fields API (available from WC 8.9+ and confirmed stable on WC 10.7.0) allows plugins to register custom fields inside the Block Checkout without modifying block internals.

- **Registration:** woocommerce_register_additional_checkout_field({ id:'upsellbay/offer-bump', label:'', location:'order', type:'checkbox', sanitize_callback:..., validate_callback:... }) registers the bump as a custom field. The 'order' location renders it below order summary - the correct visual placement for a checkout bump.
- **Rendering:** The field's label is overridden via the \_\_experimentalRegisterCheckoutFilters or slot-fill mechanism to render the full offer HTML widget (product thumbnail, headline, price, checkbox) rather than a plain text label.
- **Interaction:** On checkbox state change (Block Checkout fires an event), the upsellbay-blocks.js entry point calls /wp-json/upsellbay/v1/bump-toggle via fetch(). Cart totals are refreshed via WC Blocks' internal store dispatch(CART_STORE_KEY).invalidateResolutionForStoreSelector('getCartTotals').
- **Attribution:** The woocommerce_store_api_checkout_update_order_from_request hook writes \_ub_bump_accepted = 1 to the order before payment, ensuring woocommerce_checkout_create_order_line_item fires with the bump item present for attribution meta writing.
- **Classic checkout fallback:** The classic checkout PHP hook path (woocommerce_review_order_before_submit) is maintained independently. Detection: if is_checkout() and WC Blocks is rendering the checkout, use the block path; otherwise use the classic path.
